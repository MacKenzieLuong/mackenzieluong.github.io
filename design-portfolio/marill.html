The Problem

University students and event organizers face challenges managing event attendance:

  • Organizers: Manual check-ins are slow and error-prone, especially for large events. No easy way to track attendance trends.
  • Attendees: Paper sign-ups are tedious. Discovering campus events is difficult and scattered across platforms.
  • Event Hosts: Limited tools to promote events and engage with potential attendees before events.

Technical Challenge: Design a scalable Android application architecture that supports real-time check-ins, QR code scanning, and role-based permissions (attendees vs. organizers).

Design Challenge: Create my first UI design system that could guide 4 developers through implementation while learning industry design tools and patterns.

Competitive Analysis

Analyzed leading event and social apps to understand established patterns:

  • Eventbrite: Great discovery, but check-in process too complex for casual campus events
  • Instagram Events: Good social integration, but lacks robust check-in and analytics
  • Meetup: Strong community features, but feels too formal for student events

Design Opportunity: Combine Instagram's social ease with Eventbrite's event management, optimized for mobile-first generation.

Key Design Decisions

üì± Tab Bar Navigation

Decision: Bottom tab bar with 5 primary sections
Rationale: Familiar pattern from Instagram/Twitter. Thumb-friendly on large phones. Team agreed on technical simplicity.

üì∏ Image-First Event Cards

Decision: Large event images with minimal text overlay
Rationale: Gen Z responds to visual content. Testing showed 40% more taps on image-heavy cards vs text-heavy.

‚ö° One-Tap Check-In

Decision: QR code scan + geofence backup
Rationale: QR is fast for large events. Geofence catches users who forget to scan. Reduces check-in friction to <5 seconds.

üë• Social Features

Decision: Show friends attending, not total count
Rationale: "3 friends going" is more persuasive than "50 people going." Creates FOMO through personal connections.

üîî Smart Notifications

Decision: Event reminder 1 hour before + check-in prompt on arrival
Rationale: Balances engagement without notification fatigue. Geofence trigger feels magical, not pushy.

üìä Organizer Dashboard

Decision: Mobile-optimized analytics, not desktop clone
Rationale: Student organizers manage events on-the-go. Show key metrics (attendance rate, check-in speed) not raw data dumps.

Reflection & Learnings

‚úì What Went Well

  • Design system approach: Investing time upfront in components paid off. Developers loved having a single source of truth.
  • Playful branding: The Marill theme made the project memorable and gave us identity in a crowded event app space.
  • Mobile-first thinking: Designing for Android constraints taught me to embrace platform conventions, not fight them.
  • Team collaboration: Regular check-ins with developers prevented late-stage surprises about feasibility.

‚ñ≥ What I'd Do Differently

  • More user testing: Only tested with 8 users due to time constraints. Would do guerrilla testing on campus for faster feedback loops.
  • Accessibility earlier: Added contrast checks late in process. Should have been part of design system from day one.
  • Developer education: Team wasn't familiar with design handoff best practices. Would create onboarding doc explaining Figma Inspect and component usage.
  • Dark mode: Didn't plan for it initially, causing rework. Should have designed both themes simultaneously.

üí° Key Takeaway

This project transformed me from a designer who makes pretty screens to one who thinks in systems. Working with developers taught me that design is a conversation, not a handoff. The Marill blue theme proved that even enterprise-like apps (event management) can have personality and playfulness. Most importantly, I learned that constraints - whether technical limitations or Android's design guidelines - aren't obstacles. They're creative fuel that push designs toward elegant simplicity.
