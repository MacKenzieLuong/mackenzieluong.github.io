Project Overview

Marill is an Android application for event hosting, management, and attendee check-ins, built as a semester-long software engineering project. While this was primarily a development-focused project, I took on technical and visual design and project management responsibilities, planning the technical architecture, coordinating development, and collaborating on the high fidelity mockups. Named after the Pok√©mon Marill (a water-type!), we integrated water-inspired elements throughout the mockups and embraced a sea blue theme.

My Primary Roles: Software Architect, Project Manager, UI/UX Designer

The Problem

University students and event organizers face challenges managing event attendance:

  • Organizers: Manual check-ins are slow and error-prone, especially for large events. No easy way to track attendance trends.
  • Attendees: Paper sign-ups are tedious. Discovering campus events is difficult and scattered across platforms.
  • Event Hosts: Limited tools to promote events and engage with potential attendees before events.

Technical Challenge: Design a scalable Android application architecture that supports real-time check-ins, QR code scanning, and role-based permissions (attendees vs. organizers).

Design Challenge: Create my first UI design system that could guide 4 developers through implementation while learning industry design tools and patterns.

Target Users

Anyone with a phone wanting to join events. We designed for maximum accessibility - no demographic barriers, no tech expertise required. If you have a smartphone and want to discover or organize events, Marill is for you.

Figma Design System

As a developer designing for developers, I built a tech-forward design system that speaks their language. Rather than creating beautiful but confusing mockups, I focused on making the Figma file a functional tool for implementation.

User Story Labels on Every Screen

Approach: Labeled each screen and scenario with corresponding user stories directly on mockup frames. Each flow was clearly marked, ensuring all tasks were covered and accounted for in the UI - no separate documentation to hunt for.

Impact: Team could immediately see which user story each screen addressed, making it easy to verify completion and understand context. This labeling system ensured nothing was missed and kept everyone aligned on what was being built and why.

Working Prototype Over Specs

Approach: Created fully interactive prototype showing all navigation flows and state changes

Impact: Team could click through the app to understand connections instead of parsing flowcharts or tables - more intuitive for technical minds

üí° Bridging Tech & Design: My CS background shaped how I approached design. Instead of treating Figma as an art tool, I used it like a codebase - organized, documented, and built for collaboration. This "developer-first" design approach became my strength: I create designs that engineers love to implement.

Competitive Analysis

Analyzed leading event and social apps to understand established patterns:

  • Eventbrite: Great discovery, but check-in process too complex for casual campus events
  • Luma: Sleek, modern design with social media-heavy approach, but lacks built-in QR code functionality that we needed for our project requirements

Design Opportunity: Combine social ease with robust event management, optimized for mobile-first users with built-in QR check-in.

Design Process

Course Structure: The project had 2 major milestones requiring high-fidelity Figma mockups at each checkpoint. Since developers needed designs before they could start coding, I had to create complete mockups much earlier than typical design timelines. The team split work - half focused on mockups, half focused on technical architecture.

01

Initial Meeting (Week 1)

Reviewed project spec and design requirements. Defined user stories and split team responsibilities - design team would focus on mockups while technical team tackled architecture, diagrams, and backend planning.

02

Initial Mockup (Week 2)

Created first high-fidelity mockups based on prioritized user stories. Established core design system (colors, typography, basic components) while the other half of the team worked on technical specifications.

03

Halfway Point Mockup (Week 3-4)

Completed at least half the project mockups for Milestone 1. Focused on customer-facing features (event feed, event details, check-in flow) and left admin UI for second half. This allowed us to shift focus to technical work - diagrams, architecture documentation, and code implementation.

04

Final Mockup (Week 6-7)

Completed remaining user stories including organizer dashboard and admin interfaces. Delivered comprehensive mockup covering full application. With mockups finished, entire team could focus on code implementation and technical deliverables for final milestone.

üí° Process Learning: Designing for milestones meant I couldn't iterate slowly. This forced me to make confident decisions early and taught me to prioritize "good enough to build" over "perfect on paper."

Design System - The Marill Theme

Named after Pok√©mon Marill (a water-type!), we integrated water-inspired visual elements throughout the mockups while keeping the design clean and trustworthy.

üíß Blue Color System

Challenge: Everything was blue - we needed enough contrast to differentiate elements without relying on color alone (colorblindness accessibility)
Solution: Used saturation and brightness variations, combined with shape and size differences. Kept minimal color palette to maintain cohesion.

‚ú® Water-Inspired Elements

Subtle Details: Incorporated water-themed visual touches in mockups (water themed events, deep sea like aesthetic)
Rationale: Kept Marill theme present without losing professional trustworthiness or distracting from functionality.

üéØ Trustworthy Aesthetic

Goal: Sleek, professional look despite playful theme
Approach: Clean layouts, generous whitespace, consistent component styling. Blue theme conveys trustworthiness while maintaining approachability.

Software Architecture & Project Management

Beyond UI design, I led the technical architecture and coordinated the development team through a semester-long build cycle.

Architecture Design

Designed the Android app architecture using MVVM pattern, defined API contracts, and planned database schema for scalable event management

Sprint Planning

Led weekly sprints with task breakdown, managed GitHub project board, and prioritized features based on technical dependencies and user value

Technical Tradeoffs

Initial design had custom animations requiring weeks of work. Made pragmatic decision to use standard Android transitions to ship on time

Design-Dev Bridge

Translated design decisions into implementation specs, conducted code reviews to ensure UI fidelity, and adjusted designs based on technical constraints

Reflection & Learnings

‚úì What Went Well

  • Design system approach: Investing time upfront in components paid off. Developers loved having a single source of truth.
  • Playful branding: The Marill water-type theme made the project memorable and gave us identity in a crowded event app space.
  • Mobile-first thinking: Designing for Android constraints taught me to embrace platform conventions, not fight them.
  • Milestone-driven design: Being forced to deliver complete mockups early taught me to make confident decisions quickly.

‚ñ≥ What I'd Do Differently

  • Limited user testing: Only tested with our 6 team members due to time constraints. Should have done guerrilla testing on campus for external feedback.
  • Role clarity: Most team members wore multiple hats (design + code), which meant no one could focus deeply on one area. Would push for clearer role division.
  • No dark mode: Didn't implement dark mode - would have required too much additional work given our split responsibilities.
  • Accessibility earlier: Added contrast checks late in process. Should have been part of design system from day one, especially with our all-blue color scheme.

üí° Key Takeaway

This project transformed me from a student who can code and draw to someone who ensures that the feature requests are met, in both design and in implementation. Working with developers taught me that design is a conversation, not a handoff. The Marill water-type theme proved that even functional apps (event management) can have personality and also helped our team bond since we all loved that cute blue ball creature. Most importantly, I learned how important it is to balance the technical, design, and management aspects of a project. Everyone has a key role and contributes to the final product.
